# Run event transport (NDJSON ↔ SSE ↔ WebSocket)

Канон событий исполнения — плоский объект с полем **`type`**, валидация **`schemas/run-event.schema.json`** (**§3.7**). Транспорт (**SSE**, **WebSocket**) **не** добавляет новых значений **`type`**.

## 1. Одна строка NDJSON ↔ один кадр

| Транспорт | Форма |
|-----------|--------|
| **Worker stdout** | Одна строка = один JSON-объект (возможны префиксы вне контракта — клиент обрабатывает как сырой лог). |
| **Broker SSE** | Событие **`message`**: тело **`data:`** = та же строка, что ушла в stdout (без обёртки). События **`err`** / **`exit`** — см. `run_broker/broadcaster.py` → `stream_queue`. |
| **Broker WebSocket** | Одно **text**-сообщение = один JSON-объект **конверта** (см. §2). |

## 2. JSON-конверт WebSocket (dev broker)

Все кадры — UTF-8 JSON с полем **`runId`** (строка, UUID прогона).

### 2.1. Строка stdout воркера

```json
{
  "runId": "<uuid>",
  "channel": "stdout",
  "line": "<одна строка как в NDJSON, без завершающего \\n>"
}
```

Клиент передаёт **`line`** в тот же разбор, что и **`data:`** у **EventSource** (`parseRunEventLine` / консоль).

### 2.2. Stderr воркера

```json
{
  "runId": "<uuid>",
  "channel": "stderr",
  "payload": { "line": "..." }
}
```

`payload` совпадает с телом **SSE** `event: err`.

### 2.3. Завершение процесса воркера

```json
{
  "runId": "<uuid>",
  "channel": "exit",
  "code": 0
}
```

После **`exit`** подписка на стороне сервера снимается (как у **SSE**).

### 2.4. Входящие сообщения клиента (duplex)

Текстовый кадр от клиента — JSON:

```json
{ "type": "cancel_run", "runId": "<uuid>" }
```

Эквивалентно записи той же строки в **stdin** воркера (**`--control-stdin`**).

## 3. Сессия **`viewerToken`** (аналог pushRef, dev)

После **`POST /runs`** тело ответа содержит **`runId`** и **`viewerToken`** (непредсказуемая строка).

Подключение **WebSocket**:

`GET /runs/{runId}/ws?viewerToken=...`

Алиас query: **`pushRef`** = то же значение, что **`viewerToken`**.

Закрытие соединения до **`accept`** (реализация **`run_broker/app.py`**):

| Ситуация | WebSocket close code |
|----------|---------------------|
| Неверный или отсутствующий опциональный **`GC_RUN_BROKER_TOKEN`** (заголовок / query **`token`**) | **1008** (policy violation) |
| Неизвестный **`runId`** в пути | **4404** (логический «не найден», диапазон приложения RFC 6455) |
| Неверный / отсутствующий **`viewerToken`** / **`pushRef`** | **4401** (логический «не авторизован») |

Публичный endpoint без проверки токена **не** используется (**§39.2 п.4**). Опционально **`GC_RUN_BROKER_TOKEN`**: тот же заголовок / query **`token`**, что для HTTP (**middleware**).

**Замечание по продакшену:** query-параметры (**`viewerToken`**, **`token`**) попадают в URL и могут оказаться в логах прокси, **Referer** (при переходах) и истории браузера. Для публичного контура предпочтительны **BFF** / короткоживущая сессия (**cookie**), выдача WS URL без секрета в query, либо обмен токена **первым сообщением** после **`accept`** — с явной политикой истечения. Текущая схема рассчитана на **локальный dev** за **Vite** proxy и доверенную сеть.

**SSE** **`GET /runs/{runId}/stream`** по-прежнему без **`viewerToken`** (локальный dev; защита — только dev-token при необходимости).

## 4. Ping / pong (рекомендация для prod)

n8n ориентируется на **~60 s** keepalive. Для долгих прогонов за **nginx** рекомендуется:

- сервер: периодический **WebSocket ping** (кадр протокола) или редкий JSON **`{"channel":"ping"}`**;
- прокси: увеличить **read timeout** для location WS.

Текущий dev broker может обходиться без ping при локальном **Vite** proxy; при появлении обрывов — включить по продуктовому ТЗ.

## 5. Relay (вне этого репозитория)

Публикация событий из отдельного воркера (**Redis** / очередь) с тем же конвертом §2 — слой **Aura** (**§39.2 п.7**).

## 6. Модуль Python

**`graph_caster.run_transport`** — разбор NDJSON, упаковка **`{ runId, event }`** для валидных **`run-event`**, **`ndjson_line_from_event`**. Не тянет **Starlette** при импорте.
