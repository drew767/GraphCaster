# GraphCaster — план разработки

Порядок ориентирован на **короткие инкременты**, каждый из которых оставляет репозиторий в **собираемом/запускаемом** состоянии для своей части (Python CLI / статический UI).

## Фаза 0 — каркас (сделано)

- Дерево `third_party/graph-caster/` с `python/`, `ui/`, `schemas/`, `doc/`.
- Минимальный пакет Python: `GraphDocument.from_dict`, `GraphRunner` с событием стримом, CLI `-d/--document`.
- Пример JSON: `schemas/graph-document.example.json`.
- Заготовка `ui/package.json` + README.

## Фаза 1 — Контракт JSON v1

- Зафиксировать поля `meta`, `nodes`, `edges`, `viewport`, версию `schemaVersion`.
- Добавить **JSON-Schema** файл (по желанию рядом с `schemas/`) и тест на валидность примера. **Сделано:** `schemas/graph-document.schema.json`, pytest в `python/tests/test_schema_example.py`.
- Описать соглашения по **id пинов** (`in_default`, `out_default`, кастомные handles). **Частично:** см. корневой `README.md` → раздел «Контракт данных».
- Политика **множественных исходов** из одной ноды: порядок проверки условий, обязательный ли `else`. **Открыто:** зафиксировать явно (сейчас в раннере — «первое подходящее ребро»).

## Фаза 2 — Runner: условия и вложенность

- Заменить заглушку условий на **мини-DSL** или **JSONLogic** (один выбранный подход).
- События для **`graph_ref`**: `nested_enter` / `nested_exit` или делегирование через callback хоста.
- Защита от циклов: лимиты шагов + отчёт в консоль; опционально детектор циклов на графе.
- Юнит-тесты: линейный граф, ветвление по условию, неоднозначные исходы (ожидаемая ошибка).

## Фаза 3 — UI: скелет приложения

- Поднять **Vite + React + TS** в `ui/`.
- Разметка **3 зон**: top menu bar, center canvas, right inspector, bottom resizable console — как контейнеры даже без логики.
- Тема: токены цветов/радиусов под «Material/Apple-like».

## Фаза 4 — Canvas и ноды

- Интеграция **React Flow** (или аналог): пан/зум, кастомный node view (иконка, заголовок, описание).
- Пины: визуально слева/справа; **валидация** соединений (in↔out only).
- Сохранение/загрузка: сериализация в контракт v1, совместимая с Python `from_dict`.

## Фаза 5 — Редактор свойств и условий

- Правая панель: редактирование `data` для `task`; для `graph_ref` — выбор файла/id (адаптер к хосту).
- Редактор **условия** на ребре (текстовое поле + подсказки + валидация).

## Фаза 6 — Меню и файлы

- Новый / открыть / сохранить: через **File System Access** в браузере или мост хоста (Tauri).
- Недавние документы (локальный список в `localStorage` для веб-демо).

## Фаза 7 — Запуск и мост к Python

- Кнопка Run: в десктопе — spawn/subprocess или сокет к `graph_caster`; в веб-демо — **воспроизведение записанного лога** или опциональный backend.
- Протокол: **NDJSON** или WebSocket с теми же полями, что `GraphRunner.emit`.

## Фаза 8 — Полировка UX

- Анимации активной ноды/ребра, настройка «уменьшить эффекты».
- Отмена/повтор для движения нод и удалений.
- Локализация строк UI (при необходимости).

## Фаза 9 — Встраивание в продукт

- NPM-пакет или готовый `dist/` + документация API (`openDocument`, `onEvent`, `getSerialized()`).
- Соглашение с **Aura Desktop**: путь к `third_party/graph-caster`, возможность обновления подмодуля/копии.

## Критерии готовности MVP (сжато)

- Один JSON открывается в UI и прогоняется runner-ом с событиями в консоль.
- Два типа нод, корректные соединения пинов, условие на ребре влияет на путь.
- Сохранение восстанавливает позиции и содержимое нод.

При изменении порядка фаз обновляйте этот файл и при необходимости «Открытые вопросы» в `PRODUCT_DESIGNE.md`.
