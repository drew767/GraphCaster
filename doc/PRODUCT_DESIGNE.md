# GraphCaster — продуктовый дизайн

Дата: 2026-03-27 (обновлено: workspace, вложенные артефакты, режим Run)  
Кодовое имя: **GraphCaster**  
Архитектурный выбор: **вариант A** — UI (веб / встраивание) + **Python runner** в этом репозитории. Данные и выполнение сценария **локальны относительно экземпляра графа**; обращения к внешним сервисам (хранилища, API, **Cursor CLI**, ИИ) — через действия нод, а не через обязательный «хост приложения».

## 1. Цель продукта

Дать авторам сценариев описывать процессы как **граф документов** (ноды, направленные связи, условия), редактировать на **бесконечном полотне** и **выполнять** с обратной связью (подсветка шага, консоль). Документ — **JSON**; ссылки на другие графы — **только ссылка** на документ (без автоматического встраивания копии).

**MVP по смыслу продукта:** возможность собрать процесс **автоматической разработки с подачей промптов в Cursor CLI** (цепочка нод, данные между шагами, условия, долгие/циклические прогоны). Остальной функционал подчиняется этой цели.

## 2. Разделение UI и Python

- **UI:** геометрия, редактирование, валидация связей, консоль с полным набором операций (см. ниже).
- **Python:** загрузка документа, обход, оценка условий, жизненный цикл **Run** (контекст выполнения, артефакты), события для UI/логов.
- Встраивание в **Aura** как плагин — **позже**; контракт документа и событий проектируется так, чтобы перенос не ломал модель.

## 3. Ключевые ноды и завершение

### 3.1 Start (одна на граф)

- Специальная нода **старт** графа.
- **Нет входящих пинов** (в граф не ведёт ни одно ребро в эту ноду).
- Может иметь **параметры сценария**: в инспекторе это **обычные поля** (без входящих рёбер в саму ноду). По смыслу те же параметры дублируются как **исходящие пины**: ими можно **подключаться к входам других нод** (значения при старте Run подставляются из параметров Start).
- На графе **ровно одна** такая нода (валидатор редактора + раннера).

### 3.2 Exit (может быть много)

- Одна или несколько нод **выхода**.
- Могут иметь **параметры** (итоговые данные, статусы).
- **Успешное завершение выполнения** = достижение ноды Exit; при одном потоке выполнения сценарий **«два exit одновременно» не допускается** — срабатывает **первый достигнутый Exit**, Run завершается.

### 3.3 Прочие типы (база)

- **`task`** (или специализированные подтипы позже): произвольный **`data`** (JSON), исполнение шага сценария (в т.ч. вызов внешнего инструмента). Для нод вроде **Cursor CLI** итог шага задаётся **конфигурацией ноды**: один из режимов **успеха** — *код возврата процесса*, *разбор stdout по правилу*, *наличие/содержимое файла-маркера в репозитории* (детали полей — в контракте ноды).
- **Таймаут шага, принудительное завершение процесса и повторы (retry)** — **обязательны в MVP** для нод, запускающих внешние процессы; число повторов и бэкоф задаются в `data`/инспекторе.
- **`graph_ref`:** ссылка на другой граф по **GUID** (`graphId`). Реестр **guid → файл** строится **автосканом** выделенной папки графов (см. §5). В UI — **кнопки** (открыть граф из списка, URL в браузере); **открыть** доступно из **выпадающего меню** сверху и из **инспектора** выбранной ноды. Семантика вызова: по умолчанию **дождаться** завершения вложенного графа; переопределение — в настройках ноды/графа.

## 4. Режимы выполнения и параллелизм

- Граф **не обязан** быть открытым в редакторе во время выполнения: допускается фон и параллельная работа над другим документом. **Встроенный планировщик (будильник / расписание) в продукте на первом этапе не делаем** — позже или внешний вызов «старт Run».
- **Базовая версия:** один **логический поток** выполнения (один активный «луч» в графе).
- **Идея на будущее:** специальная нода «разветвление → слияние» (несколько путей, затем точка ожидания). Визуал и встраивание в модель — **не зафиксированы**; к **MVP не относятся**.
- Несколько **разных графов** могут выполняться **независимо и параллельно**: у каждого прогона свой **Run** (изолированный контекст данных и событий). Архитектурное решение см. `DEVELOPMENT_PLAN.md` (координация сессий, при необходимости отдельный процесс на Run для устойчивости к сбоям CLI).
- **Пока Run этого документа активен — редактирование графа запрещено** (полотно и структура в режиме только чтения / заблокированы операции правки — конкретика UX: блокировать save изменений структуры или весь редактор кроме просмотра и стопа).

## 5. Контекст данных и типы

- Выполнение **локальное**: переменные и результаты нод живут в **контексте итерации Run**.
- Выход ноды: **скаляры** (int, bool, string, float) и **крупные объекты** (изображение, аудио, видео, документы, ссылки и т.д.) — как минимум **ссылки/дескрипторы + метаданные** в JSON (бинарь не обязан встраиваться в документ графа).
- Поток данных: результат одной ноды доступен следующим (и ветвлениям по условиям).
- **Workspace: папка графов и артефакты (MVP):**
  - Выделенная **папка графов** в workspace (рабочее имя по умолчанию, напр. **`graphs/`**, уточняется в настройках): все **сохранения JSON** только туда, **автоматически**, **без** диалога выбора пути. **Загрузка** списка графов — **автоскан** этой папки; открытие документа — из **меню «Открыть»** (верхняя полоса) и кнопки **«Открыть»** в **инспекторе** выбранной ноды (навигация по ссылкам в графе).
  - **Уникальность `graphId`:** двух графов с одним и тем же **`graphId`** в workspace **быть не должно**. Если такой **graphId** уже есть в проиндексированных файлах — **нельзя сохранить** новый документ с этим id и **нельзя переименовать**/назначить id так, чтобы нарушить уникальность (жёсткая блокировка в редакторе).
  - **Каноническая идентичность** — **`graphId`** в `meta`; имя файла на диске может меняться, id **нет**.
  - **Артефакты выполнения** (большие файлы, вывод нод): корень по умолчанию **`runs/`** в workspace; структура **`runs/<graph_guid>/<run_datetime>/`** для **корневого** запуска (см. вложенность ниже).
  - **Вложенный граф** при выполнении **не** заводит для этого же логического прогона отдельное дерево «своих» артефактов по своему `graph_guid` на верхнем уровне: **все записи** идут в каталог Run **корневого (верхнего) родителя** — тот самый **`runs/<root_graph_guid>/<run_datetime>/`**. Раннер **неявно передаёт** вниз по стеку `graph_ref` ссылку на **root run** (artifact root); при создании файла запрос поднимается к корню. Внутри корневой папки допускаются подпапки по нодам/вложениям (деталь реализации).
  - **Очистка артефактов:** кнопка **«Очистить»** (артефакты/история прогонов — уточнить область в UX). В **настройках** — отображение **объёма на диске**, занятого артефактами **по каждому графу** (сумма по `runs/<graph_guid>/`), чтобы видеть «сколько съел» конкретный граф.
- Офлайн: редактор и раннер работают без сети **пока ноды не требуют внешних сервисов**.

## 6. Условия на рёбрах (уровни)

1. **Простые:** галочки, сравнения (больше/меньше/равно), предикаты над примитивами из контекста.
2. **Средняя сложность:** строковые операции (подстрока, парсинг числа и т.п.) — формализованный набор операций без произвольного кода.
3. **ИИ-ветвление (структурированный JSON):** после выполнения ноды раннер (или адаптер) формирует **запрос в виде JSON**: передаётся **результат выполнения ноды** (срез контекста / `last_node_output`, без лишних МБ) и **список исходящих связей** — у каждой связи **номер (индекс)** и **описание** (человекочитаемо: куда ведёт, зачем эта ветка). Опционально — короткая **инструкция автора** к решению. **Ответ модели** — тоже **строго структурированный JSON**, например поле вроде **`choiceIndex`** / **`selectedConnectionIndex`** с целым **1…N**, однозначно указывающим выбранную связь из переданного списка. Невалидный ответ, индекс вне диапазона — политика ошибки (повтор, fallback, стоп Run) задаётся в настройках ноды/раннера. **Провайдер** (HTTP к своему endpoint, локальная модель и т.д.) и ключи — отдельно от этого контракта.
4. **Безопасность выражений:** избегать произвольного `eval`; для произвольного кода — только с явной песочницей/политикой (прорабатывается вместе с пунктом об ошибках выполнения).
5. **Передача контекста в ИИ (секреты):** маскирование токенов/секретов в payload перед вызовом провайдера — **на рассмотрении** (открытый предмет), см. план.

Пересечение нескольких истинных исходов из одной ноды:
- **Рантайм:** перебор **исходящих рёбер в том порядке, в каком они перечислены в JSON**; выбирается **первое** с истинным условием (или без условия как «всегда да» по текущим правилам).
- **Редактор:** если валидатор обнаруживает, что **более одного** ребра может оказаться истинным при текущем контексте (статический анализ ограниченный) или есть **риск перекрытия** — подсветка блока рёбер **жёлтым** + иконка **предупреждения** (восклицательный знак в треугольнике). Это **не блокирует** сохранение документа и редактирование — только информирует.

## 7. Ошибки выполнения

- Политика при сбое шага (стоп / переход по «ошибочному» ребру / повтор / делегирование) — **открытый предмет**; прорабатывается в процессе разработки совместно с UX и сценариями Cursor CLI.

## 8. Метаданные документа

- Сохранять **имя автора** (и при необходимости прочие мета-поля) в документе.
- Для **запуска** автор **не валидируется** (нет обязательной криптографической проверки).

## 9. Версионирование

- Заложить в модель документа и схему **фундамент под версии** (идентификаторы ревизий, совместимость). **Полную** систему версий **не** реализовывать в первой волне.

## 10. UI/UX (требования к MVP и консоль)

- **Редактор:** мультивыбор нод, групповое удаление, буфер обмена — **в MVP**.
- **Отмена / повтор** — **нужны** (движение нод, соединения, значимые правки).
- **Мини-карта**, **поиск по нодам**, **фильтр по типу** — **в MVP**.
- **Добавление нод** через **контекстное меню** с **поиском** по типу/шаблону.
- **Консоль:** не только вывод; **уровни**, **фильтры**, **экспорт**, **очистка**.
- **Локализация:** с **первого** релиза — **русский** и **английский**; **язык по умолчанию** — от **системной локали** (с возможностью ручного переключения в настройках).

## 11. Внешний вид (кратко)

Без изменений к базовому направлению: бесконечное полотно, меню сверху, динамический правый инспектор ~1/5–1/6, нижняя ресайзабельная консоль; ноды с иконкой, названием, описанием; пины слева вх., справа исх.; правила соединений; анимация активного шага (с опцией упрощения).

## 12. JSON и совместимость

- Документ графа — **переносимый JSON**; `schemaVersion` для эволюции.
- Поля нод **`data`** — **произвольный JSON** (строгость валидации — по мере готовности типов нод и редактора).
