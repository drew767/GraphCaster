# GraphCaster — продуктовый дизайн

Дата: 2026-03-27  
Кодовое имя: **GraphCaster**  
Архитектурный выбор: **вариант A** — тонкий UI (веб / встраивание) + **Python runner** как отдельный пакет в этом дереве.

## 1. Цель продукта

Дать авторам сценариев **один и тот же** способ описывать логику как **граф документов** (ноды и направленные связи с условиями), редактировать его на **бесконечном полотне** и **выполнять** с визуальной обратной связью (подсветка активной ноды/ребра, лог в консоли). Документ графа — **переносимый JSON**; вложенные графы — **отдельные документы**, на которые ссылается нода типа «вложенный граф».

## 2. Почему разделены UI и Python

- **Лёгкость и переносимость:** интерпретатор графа не тянет за собой тяжёлый UI-стек; его можно вызывать из CLI, тестов, десктоп-хоста (Tauri), сервиса.
- **Один UI — два способа показа:** статическая веб-сборка (браузер, WebView) и встраивание через iframe/SDK событий; поведение редактора одинаковое.
- **Веб без обязательного Python в браузере:** в чистом браузере возможны **редактирование и анимация «прогона» по записанному логу**; **живое выполнение** в варианте A предполагает хост с Python рядом (десктоп) или бэкенд — контракт остаётся одним (JSON документ + поток событий).

Итог: **UI отвечает за геометрию, редактирование и отображение**, **Python — за семантику шага и переходов** (в MVP — упрощённо, далее расширяется).

## 3. Внешний вид и компоновка

### 3.1 Стиль

- **Современный «мягкий» UI:** скругления **8–16 px** на карточках нод и панелях, много **воздуха**, нейтральная палитра (светлая тема по умолчанию), **системные или переменные шрифты** в духе Material / Apple HIG: чёткая иерархия заголовок / подзаголовок / вторичный текст.
- **Без визуального шума:** линии рёбер тонкие, акцентный цвет только для **активного** узла/ребра и **фокуса** инспектора.

### 3.2 Верхняя зона (меню-бар)

Полоса **на всю ширину**, сверху. Содержит **выпадающие меню** (как в классическом desktop app):

- **Файл:** новый граф, открыть, сохранить, сохранить как, недавние, закрыть документ.
- **Правка:** отмена/повтор (по мере появления стека), дублировать ноду, удалить, настройки сетки/привязки (хуки позже).
- **Вид:** масштаб, сброс вида, показать/скрыть консоль, показать/скрыть правую панель.
- **Запуск:** старт/стоп сценария, шаг (если будет пошаговый режим), политика при ошибке.

Кнопки **не «россыпью»**, а сгруппированы по смыслу внутри dropdown — так проще масштабировать количество действий.

### 3.3 Основная рабочая область (canvas)

- **Бесконечное полотно:** панорамирование перетаскиванием (space+drag или средняя кнопка — уточняется в UX), зум колёсиком / жестами.
- **Сетка:** едва заметная точечная или линейная сетка (опционально вкл/выкл), чтобы ориентироваться без утяжеления вида.

### 3.4 Ноды

Каждая нода — **карточка** со:

- **Иконкой** (если задана в `data`),
- **Названием** (короткий заголовок),
- **Кратким описанием** (1–2 строки вторичным цветом).

**Пины:**

- **Слева:** только **входные** (targets для рёбер).
- **Справа:** только **исходящие** (sources).

**Правила соединений (жёстко в UX и валидации):**

- Выход **нельзя** соединить с выходом; вход **нельзя** со входом.
- Область исходящих пинов **кликабельна целиком** (удобный hit-target), чтобы начать ребро; завершение — на входном пине целевой ноды.

**Анимация состояния выполнения:**

- На этапе прогона подсвечиваются **текущая нода** (обводка / мягкое свечение) и **ребро**, по которому произошёл переход.
- Допустимы лёгкие **CSS transition** или отдельный слой оверлея над canvas — всё должно быть **отключаемо** для слабых машин (настройка «упрощённая анимация»).

### 3.5 Правая панель (инспектор / тулбар)

- Ширина по умолчанию **~1/5–1/6** окна; **ресайз** за левую границу.
- **Динамическое наполнение:** при выборе ноды — свойства типа, заголовок, описание, ссылка на вложенный документ; при выборе ребра — редактор **условия**; при пустом выборе — сводка по документу или подсказки.
- При смене выделения содержимое **заменяется**, без «хвостов» от предыдущего объекта.

### 3.6 Нижняя консоль

- **Тулбар с логами:** высота регулируется (**drag** верхней границы), можно **свернуть** в минимальную полосу.
- Вывод **событий выполнения** в хронологическом порядке: время, тип события, краткая полезная нагрузка (id ноды, id ребра, сообщение).
- Отдельные уровни: info / warn / error (цветом и иконкой, без перегруза).

## 4. Модель данных (документ графа)

### 4.1 Документ как единица хранения

- Один файл JSON = **один граф** с уникальным `graphId` (в пределах рабочего пространства пользователя или проекта).
- **Позиции нод** и **весь payload** нод сохраняются **полностью** (чтобы после загрузки редактор выглядел так же).
- **Версия схемы** `schemaVersion` для миграций вперёд.

### 4.2 Типы нод (MVP)

1. **`task`** — атомарная задача: заголовок, описание, произвольные поля в `data` (параметры сценария).
2. **`graph_ref`** (продуктовое имя в UI: «Вложенный граф» / «Ссылка на граф») — ссылка на **другой JSON-документ** (путь, id в каталоге или URI — деталь хранилища определяется хостом). Семантически: при исполнении раннер может **загрузить** вложенный документ и выполнить как подпрограмму **или** отдать хосту (политика встраивания).

Вложенный граф **структурно такой же**, как родитель; автор может быть другим — это вопрос **происхождения файла**, не формата.

### 4.3 Рёбра и условия

- Ребро соединяет **конкретный исходящий пин** с **конкретным входным**.
- **Условие** — строка или структурированное выражение (в плане: простой DSL или JSONLogic); в MVP допускается **минимальный** набор: пусто / всегда истина / предопределённые ключи контекста.
- Несколько исходящих из одной ноды: раннер выбирает **первое подходящее** по правилу продукта (документируется в DEVELOPMENT_PLAN: явный приоритет, «else», запрет пересечений).

## 5. Поток выполнения (логика)

1. Загрузка JSON → валидация схемы (минимальные инварианты: уникальные id, существование концов рёбер, типы нод).
2. Старт с **выбранной** или **единственной** входной ноды (политика старта фиксируется в плане релиза).
3. Для каждой ноды: события **`node_enter` → `node_execute` → `node_exit`**.
4. Выбор исходящего ребра с учётом **контекста** и **условий**; событие **`edge_traverse`**.
5. Для **`graph_ref`:** событие уровня «вход во вложенный граф» / делегирование хосту — по выбранной стратегии в плане.
6. Завершение: **`run_end`** или **`error`**.

События дублируются в **консоль UI** и могут стримиться по каналу к хосту.

## 6. Почему JSON

- **Человекочитаемость**, диффы в Git, простые **бэкапы**.
- **Независимость** от конкретного UI-фреймворка: тот же файл в CLI и в редакторе.
- **Версионирование** через `schemaVersion` без ломки старых файлов (постепенные миграции).

## 7. Нефункциональные ожидания

- **Лёгкий рантайм:** Python без тяжёлых зависимостей на первом этапе; опциональные зависимости под расширенный движок выражений — отдельным extra.
- **Кроссплатформенность:** UI — там, есть ли WebView/браузер; runner — где есть Python 3.10+.
- **Безопасность (на будущее):** выполнение пользовательских выражений из графа не должно превращаться в произвольный `eval` без песочницы — в дизайне заложено **узкое DSL/API контекста**.

## 8. Границы MVP vs дальше

**MVP:** два типа нод, полотно, пины/рёбра, сохранение позиций, простые условия, консоль, базовый runner с потоком событий, заготовка UI.

**Далее:** collab, плагины типов нод, строгая типизация пинов, отладчик, remote runner, политики параллелизма, библиотека готовых графов.
